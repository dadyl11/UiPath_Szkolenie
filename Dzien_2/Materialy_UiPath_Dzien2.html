Wprowadzenie — email w RPA

Automatyzacja poczty to jeden z najczestszych scenariuszy RPA w biznesie. Roboty wysylaja raporty, pobieraja zamowienia z emaili, parsuja dane z zalacznikow, odpowiadaja na typowe zapytania. UiPath obsluguje email przez standardowe protokoly internetowe — nie potrzebuje GUI klienta pocztowego.

Trzy protokoly pocztowe

SMTPWysylanie465 lub 587Simple Mail Transfer Protocol — "kurier" dostarczajacy listySend SMTP Mail Message
IMAPOdbieranie993Internet Message Access Protocol — "skrzynka pocztowa na serwerze"Get IMAP Mail Messages
POP3Odbieranie995Post Office Protocol — "pobierz i usun z serwera"Get POP3 Mail Messages
IMAP vs POP3 — ktorej uzywac?
CechaIMAPPOP3
Wiadomosci na serwerzeZostaja (synchronizacja)Usuwane po pobraniu (domyslnie)
Wiele urzadzenTAK — emaile dostepne wszedzieNIE — po pobraniu znikaja z serwera
FolderyTAK — obsluguje foldery (Inbox, Sent, wlasne)NIE — tylko Inbox

Gmail — wymagane App Password

Gmail od 2022 roku blokuje logowanie "mniej bezpiecznymi aplikacjami". Aby UiPath mogl wysylac emaile przez Gmail:

  1. Wlacz weryfikacje dwuetapowa (2FA) na koncie Google
  2. Wygeneruj App Password: Google Account → Security → App Passwords
  3. Uzyj wygenerowanego hasla (16 znakow) zamiast hasla do konta
Wazne: App Password to jednorazowy klucz — wygeneruj osobny dla kazdego robota. Nigdy nie uzywaj glownego hasla do konta!

Bezpieczenstwo hasel w UiPath

Dlaczego nie trzymamy hasel w workflow?

Wyobraz sobie scenariusz: publikujesz robota do Orchestratora, a w workflow masz Password = "mojeHaslo123". Kazdy, kto ma dostep do projektu, widzi haslo w czystym tekscie. Co gorsza — haslo moze trafic do logow, do repozytorium git, do kopii zapasowych.

Hierarchia bezpieczenstwa hasel w UiPath
PoziomMetodaBezpieczenstwoKiedy uzywac
1 (najgorszy)String w workflowWidoczne wszedzie!NIGDY w produkcji
2Windows Credential ManagerLokalne na maszynieRozwoj/testy
3 (najlepszy)Orchestrator Credential AssetSzyfrowane, centralneProdukcja
String vs SecureString
TypStringSecureString
PrzechowywanieJawny tekst w pamieciZaszyfrowany w pamieci
W logachWidoczne (jesli logowane)Ukryte (wyswietla *****)
W Type IntoWlasciwosc TextWlasciwosc SecureText
Konwersja—New NetworkCredential("", secureStr).Password (ostroznie!)
Zasada: Hasla zawsze przechowuj jako SecureString. W Type Into dla pola hasla zawsze uzywaj SecureText, nigdy Text.
Zrodla credentials

1. Windows Credential Manager (lokalnie):

  • Panel sterowania → Menedzer poswiadczen → Poswiadczenia systemu Windows
  • Aktywnosc: Get Secure Credential (Target = nazwa poswiadczenia)
  • Zwraca: Username (String) + Password (SecureString)
  • Ograniczenie: dziala tylko na maszynie, na ktorej jest zapisane

2. UiPath Orchestrator (centralnie):

  • Orchestrator → Assets → Credential
  • Aktywnosc: Get Credential (AssetName = nazwa asseta)
  • Zwraca: Username (String) + Password (SecureString)
  • Zaleta: centralne zarzadzanie, dziala na kazdej maszynie z polaczeniem

Obiekt MailMessage — "anatomia" emaila w UiPath

Gdy pobierasz emaile przez Get IMAP Mail Messages, dostajesz liste obiektow MailMessage. Kazdy obiekt to jeden email z pelnymi metadanymi:

WlasciwoscTypOpisPrzyklad
.SubjectStringTemat emaila"Zamowienie 12345"
.BodyStringTresc emaila (tekst)"Dzien dobry..."
.From.AddressStringAdres nadawcy"jan@firma.pl"
.From.DisplayNameStringNazwa nadawcy"Jan Kowalski"
.DateAsDateTimeDateTimeData i godzina2024-01-15 10:30
.AttachmentsCollectionLista zalacznikow(obiekty Attachment)
.ToCollectionLista odbiorcow(obiekty MailAddress)
.CCCollectionLista CC(obiekty MailAddress)
.IsReadBooleanCzy przeczytanyTrue / False
Przydatne metody na MailMessage
' Czy temat zawiera slowo?
currentMail.Subject.Contains("Zamowienie")     → True/False

' Ostatnie slowo tematu (np. numer zamowienia)
currentMail.Subject.Split(" ").Last()          → "12345"

' Data w formacie polskim
currentMail.DateAsDateTime.ToString("dd.MM.yyyy")  → "15.01.2024"

' Adres nadawcy
currentMail.From.Address                        → "jan@firma.pl"

' Ile zalacznikow?
currentMail.Attachments.Count                   → 2

Budowanie sciezek do plikow i tresci emaila

Budowanie sciezek do plikow — dobre vs zle praktyki
' ZLE — hardkodowana sciezka (nie zadziala na innej maszynie!):
str_sciezka = "C:\Users\Jan\Documents\UiPath\Projekt\Produkty.xlsx"

' LEPSZE — sciezka wzgledna:
str_sciezka = Environment.CurrentDirectory + "\Produkty.xlsx"

' NAJLEPSZE — Path.Combine (bezpieczne laczenie):
str_sciezka = Path.Combine(Environment.CurrentDirectory, "Produkty.xlsx")
Budowanie tresci emaila
' Sposob 1 — konkatenacja (malo czytelna przy dlugich tresciach):
str_body = "Czesc" + vbNewLine + "Raport z dnia: " + Now.ToShortDateString + vbNewLine + "Pozdrawiam"

' Sposob 2 — String.Format (czytelniejsza):
str_body = String.Format("Czesc{0}Raport z dnia: {1}{0}Pozdrawiam", vbNewLine, Now.ToShortDateString)
Wzorzec "Filtruj → Parsuj → Raportuj"

To jeden z najczestszych wzorcow w RPA:

  1. Pobierz dane — z emaila, strony www, pliku, API
  2. Filtruj — wybierz tylko to, co Cie interesuje (.Contains(), IF, Filter Data Table)
  3. Parsuj — wyciagnij konkretne informacje (.Split(), Regex, substring)
  4. Raportuj — zapisz wynik do Excela, wyslij emailem, wstaw do systemu
[Get IMAP Mail]  →  [For Each + IF]  →  [Split/Parse]  →  [Build DT + Write Range]
   POBIERZ            FILTRUJ             PARSUJ              RAPORTUJ

Modul 5 — Data Scraping

Wprowadzenie — automatyczne pobieranie danych ze stron www

Data Scraping to technika automatycznego wyodrebniania danych strukturalnych (tabelarycznych) ze stron internetowych. Zamiast recznie kopiowac dane wiersz po wierszu, robot pobiera cala tabele w kilka sekund.

Typowe scenariusze w biznesie:

  • Pobieranie kursow walut z NBP
  • Scrapowanie wynikow wyszukiwania (firmy, produkty, oferty)
  • Pobieranie danych z rejestrow publicznych (CEIDG, KRS, GUS)
  • Monitorowanie cen konkurencji
  • Zbieranie danych kontaktowych

Kluczowe aktywnosci

AktywnoscOpisKiedy uzywac
Extract Table Data (NExtractData)Pobieranie danych tabelarycznych ze stronGdy dane sa w formie tabeli (wiersze + kolumny)
Get TextPobieranie tekstu z konkretnego elementuGdy potrzebujesz jedna wartosc (np. naglowek, cene, date)
Get AttributePobieranie atrybutu elementu HTMLGdy potrzebujesz href, src, value, class
Extract Table Data — jak to dziala pod spodem?
  1. Wskazujesz pierwszy element w tabeli (np. pierwsza komorke w pierwszym wierszu)
  2. UiPath analizuje strukture HTML i probuje automatycznie rozpoznac:
    • Gdzie zaczynaja sie wiersze (<tr>)
    • Gdzie sa komorki (<td>)
    • Gdzie sa naglowki (<th>)
  3. Wskazujesz drugi element (kolejna komorka w tym samym wierszu) — UiPath potwierdza wzorzec
  4. Konfiguracja: nazwy kolumn, typy, URL-e, limit rekordow
  5. Wynik: DataTable z danymi

Paginacja — pobieranie danych z wielu stron

Wiele stron internetowych wyswietla dane na wielu stronach (np. 10 wynikow na strone). Extract Table Data obsluguje automatyczna paginacje:

ParametrOpisWartosc
MaximumResultsMaksymalna liczba rekordow do pobranianp. 50, 100
LimitExtractionToLimit typu"Rows"
NextPageSelectorSelektor przycisku "Nastepna strona"(konfigurowany w wizard)
Jak skonfigurowac paginacje
  1. W oknie Extract Table Data kliknij "Indicate Next Page"
  2. Wskaz przycisk "Nastepna" / "Next" / ">" na stronie
  3. UiPath automatycznie bedzie klikal ten przycisk po pobraniu danych z biezacej strony
  4. Pobieranie konczy sie gdy: osiagniety MaximumResults LUB przycisk "Nastepna" nie istnieje
Uwaga: Paginacja wymaga stabilnego selektora przycisku "Nastepna strona". Jesli selektor zmienia sie miedzy stronami, uzyj Fuzzy.

Try-Catch — obsluga bledow

Dlaczego bledy sa nieuniknione w RPA?

W przeciwienstwie do zwyklego programowania, roboty RPA operuja na zywych systemach: strony sie laduja wolno, pliki sa blokowane, elementy UI zmieniaja sie po aktualizacji, siec jest niestabilna. Dlatego obsluga bledow (Exception Handling) to nie "bonus" — to koniecznosc.

Struktura Try-Catch
┌─────────────────────────────────┐
│          TRY                    │
│  Kod, ktory MOZE wygenerowac   │
│  blad (np. klikniecie, odczyt) │
└────────────┬────────────────────┘
             ↓ (jesli blad)
┌─────────────────────────────────┐
│         CATCH                   │
│  Co zrobic gdy blad WYSTAPI:    │
│  - zaloguj blad                 │
│  - powiadom uzytkownika         │
│  - wykonaj akcje naprawcza      │
└────────────┬────────────────────┘
             ↓ (zawsze)
┌─────────────────────────────────┐
│        FINALLY                  │
│  ZAWSZE wykonywany:            │
│  - zamknij polaczenie           │
│  - zwolnij zasoby              │
│  - wyczysc zmienne tymczasowe  │
└─────────────────────────────────┘
Bloki Try-Catch
  • Try — kod, ktory moze wygenerowac blad
  • Catch — co zrobic gdy blad wystapi (log, powiadomienie, akcja naprawcza). Mozesz miec wiele blokow Catch dla roznych typow wyjatkow
  • Finally — zawsze wykonywany (czyszczenie, zamkniecie polaczen) — niezaleznie czy byl blad czy nie
Kiedy uzywac Try-Catch
ScenariuszDlaczego?Przyklad

Polaczenia siecioweSerwer moze nie odpowiadac, timeoutHTTP Request, Get IMAP
Automatyzacja UIElement moze nie byc dostepny, zmienil sie selektorClick, Type Into, Get Text
Parsowanie danychDane moga miec nieoczekiwany formatcint(), DateTime.Parse
Operacje na emailSerwer SMTP moze odrzucic polaczenieSend SMTP Mail Message

UI Explorer — narzedzie do analizy selektorow

UI Explorer to narzedzie do "rentgenowania" elementow UI — pokazuje co robot "widzi" na ekranie i jakie atrybuty ma kazdy element.

Kiedy go uzywac
  • Selektor automatycznie wygenerowany przez UiPath nie dziala
  • Chcesz zrozumiec strukture UI (jakie atrybuty ma element?)
  • Potrzebujesz dynamicznego selektora (musisz wiedziec, ktory atrybut jest zmienny)
  • Debugujesz problem z kliknieciem na niewlasciwy element
Mozliwosci
  • Przegladanie hierarchii elementow (drzewo UI)
  • Testowanie selektorow (Validate → sprawdz czy selektor znajduje element)
  • Dodawanie/edytowanie atrybutow selektora
  • Porownywanie selektorow (dwa elementy obok siebie)
Jak otworzyc UI Explorer
  1. W UiPath Studio: Design → Launch UI Explorer (lub Ctrl+F6)
  2. W aktywnosci (np. Click): kliknij selektor → "Open in UI Explorer"

Retry Scope vs Delay — dlaczego Retry jest lepszy
' ZLE — Delay (zgadywanie czasu):
Delay 5 sekund    ← co jesli strona laduje sie 2s? Marnujesz 3s.
Click "Login"      ← co jesli strona laduje sie 10s? Blad!

' DOBRE — Retry Scope (inteligentne ponawianie):
Retry 3 razy, co 2 sekundy:
  Click "Login"
  Condition: Element "Dashboard" Exists

Retry Scope probuje akcje, sprawdza warunek, i jesli niespelniony — czeka i probuje ponownie. Lepsze niz Delay bo:

  1. Nie marnuje czasu (jesli zadziala za pierwszym razem)
  2. Daje szanse na sukces (jesli pierwsza proba sie nie uda)
  3. Kontrolowany limit prob (nie czeka w nieskonczonosc)
